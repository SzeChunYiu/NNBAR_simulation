## Goal

Implement three foundational modules that the rest of the rebuild
depends on:

### Module 1 — realism audit (plan 01 §4)

`nnbar_reconstruction/audit/realism.py` plus
`nnbar_reconstruction/_realism.py` for decorators.

- AST walker that enumerates every parquet-column read in
  `nnbar_reconstruction/`.
- Resolves column names against plan 09 classification.
- Walks up to the enclosing function and inspects its decorators.
- Emits a violation when a Class B column is read inside a function
  lacking `@validation_only`, `@diagnostic_only`, or `@labeling`.
- CLI: `python -m nnbar_reconstruction.audit.realism --schema
  <plan-09-yaml> --source nnbar_reconstruction/ --json <out>
  --fail-on-violation`.
- Exit codes per plan 01 §4: 0 / 1 / 2.

Tests: `tests/test_realism_audit.py`. Worked example: a
deliberate-violation fixture under `tests/fixtures/realism/` that
reads `Track_ID` outside any decorator and is expected to raise.
A second fixture decorated with `@validation_only` that reads the
same column and is expected to pass.

### Module 2 — dataset registry (plan 03)

`nnbar_reconstruction/registry/` package.

- Manifest schema validator (plan 03 §3).
- Index reader/writer (plan 03 §4).
- Status-transition state machine (plan 03 §6) with freeze-acceptance
  checks (plan 03 §6.1–§6.6).
- SHA-256 hash sealing (plan 03 §7).
- Decision-log integration helper (plan 03 §9).

Tests: `tests/test_registry_integrity.py`. Verify hash mismatch is
detected; verify the freeze acceptance criteria are enforced;
verify the state-transition machine refuses illegal transitions.

### Module 3 — statistics (plan 04)

`nnbar_reconstruction/statistics/` package, one file per estimator
per the 500-line rule:

- `bootstrap.py` (plan 04 §2) — deterministic seed binding via
  `sha256(dataset_id || plan_id || "bootstrap")[:8]`.
- `jackknife.py` (plan 04 §3) — block jackknife.
- `wilson.py` (plan 04 §4) — Wilson 68% and 95% intervals.
- `feldman_cousins.py` (plan 04 §5) — F-C upper limit, with the
  worked example: `n_obs=0, mu_b=0` ⇒ `mu_s ≤ 2.44` at 90% C.L.

Tests: `tests/test_statistics.py`. One test per estimator that
exercises the worked example named in the plan section (e.g. F-C
asserts `2.44 ± 0.01` for the named case).

## Working rhythm (per iteration)

1. Re-read `docs/parallel-sessions.md` (in the orchestration repo)
   and this file (also in the orchestration repo).
2. Re-read the plan section your iteration targets.
3. Pick **one module file** to implement (≤ 500 lines).
4. Write the module + its test file in the same iteration.
5. Run `pytest tests/ -x --tb=short`. Must pass.
6. `git add nnbar_reconstruction/<module> tests/<test>`.
7. `git commit -m "feat(<area>): implement <feature>" -m "Plan: 01|03|04" -m "Lane: L3"`.
8. **No `merge.sh`.** Commits stay on `lane/L3-foundations-code`.
   The user reviews before merging into NNBAR_Detector master,
   because master is dirty with in-flight rebuild work that L3
   must not collide with.
9. Continue.

## Decision-log entries

If any implementation choice in a module makes a methodology decision
beyond what its plan section already locks (e.g. picking a specific
truncation strategy for a numeric estimator, or deciding the JSON
schema for the audit output), draft a `DEC-YYYY-MM-DD-N` entry in the
**orchestration repo** under `docs/governance/DECISION_LOG.md`. Note
this in your commit message.

(L3 commits in the NNBAR_Detector repo, but DEC entries go in the
orchestration repo because that's where plan 05's decision log
lives. Surface this in your commit message and the user will
manually add the DEC entry on the orchestration side.)
