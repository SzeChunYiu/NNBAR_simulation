# G4GPU: A General-Purpose GPU Transport Extension for Geant4

## Vision

Build **G4GPU** — an open-source GPU acceleration framework for Geant4 that any
high-energy physics experiment can adopt. Not another EM-only plugin (that's Celeritas/
AdePT). G4GPU accelerates ALL particle types, ALL physics processes, and works WITHOUT
requiring external dependencies like VecGeom.

NNBAR cosmics are the proving ground. The architecture is designed so other experiments
can plug in their own geometry and physics tables and get GPU speedup immediately.

This is publishable at a HEP computing conference (CHEP, NSS/MIC) and potentially
as a Geant4 collaboration contribution.

## Core Architecture: The G4GPU Framework

Design all five components. For each: class diagram, data layout, GPU/CPU boundary.

---

### Component 1: Track Interceptor (`G4GPUTrackingManager`)

Inherits `G4VTrackingManager`. Registered per particle species via
`G4ParticleDefinition::SetTrackingManager()`.

```
HandOverOneTrack(G4Track* track):
    → Convert G4Track (AoS) to SoA entry in GPU track buffer
    → If buffer full: flush to GPU immediately
    → Else: return (batch accumulating)

FlushEvent():
    → Flush remaining tracks to GPU
    → Wait for GPU completion
    → Convert GPU results (SoA) back to G4Track secondaries
    → Push secondaries into G4EventManager stack
```

Key design questions to answer:
- What is the optimal batch size? (memory bandwidth vs. launch overhead tradeoff)
- How to handle tracks generated by OTHER tracks mid-event? (recursive batching)
- Thread safety: multiple G4 MT threads feeding one GPU — how to synchronize?

---

### Component 2: SoA Track Buffer (`G4GPUTrackBuffer`)

Geant4's `G4Track` is AoS (Array of Structs) — terrible for GPU coalescing.
The buffer converts to SoA (Struct of Arrays) for GPU efficiency.

```cpp
struct G4GPUTrackBuffer {
    // Kinematic state (SoA layout for coalesced GPU reads)
    float* pos_x, *pos_y, *pos_z;      // position (mm)
    float* dir_x, *dir_y, *dir_z;      // direction (unit vector)
    float* kinetic_energy;              // MeV
    float* global_time;                 // ns

    // Particle identity
    int*   pdg_code;                    // PDG particle ID
    int*   material_id;                 // index into material table
    int*   volume_id;                   // index into geometry volume table
    int*   track_id;
    int*   parent_id;

    // Status
    int*   status;                      // alive/stopped/killed
    int    n_tracks;                    // current fill level
    int    capacity;                    // buffer size
};
```

Design the host↔device transfer strategy:
- Pinned memory for zero-copy? Or explicit cudaMemcpyAsync?
- Double buffering: fill buffer N+1 on CPU while GPU processes buffer N?

---

### Component 3: GPU Physics Engine (`G4GPUPhysicsKernel`)

The core CUDA kernel. One thread per track. Advances each track by one step.

```
__global__ void StepKernel(
    G4GPUTrackBuffer* tracks,
    G4GPUHitBuffer*   hits,
    G4GPUSecondaryBuffer* secondaries,
    G4GPUPhysicsTables* tables,     // cross-sections, preloaded from Geant4
    G4GPUGeometry*    geometry,
    curandState*      rng,
    int n_tracks
)
```

For each track (one thread):
1. **Get step limit from geometry**: distance to next boundary (see Component 4)
2. **Get step limit from physics**: sample interaction length for each active process
   - Winner is shortest mean free path (same logic as G4SteppingManager)
3. **Advance track**: move to `min(geo_limit, physics_limit)`
4. **Apply physics process**: update kinematics, generate secondaries if needed
5. **Record hits**: if track is in a sensitive volume, write to hit buffer
6. **Update status**: killed/alive/boundary-crossed

**Physics processes to implement (design the math for each):**

For muons (priority for cosmics):
- Ionization: Bethe-Bloch mean loss + Landau straggling via curand
- Multiple Coulomb scattering: Highland formula angular deflection
- Bremsstrahlung: sampling from tabulated differential cross-sections
- Muon nuclear interaction: approximate total cross-section, delegate secondary to CPU

For EM shower (if not using Celeritas):
- Compton: Klein-Nishina sampling
- Photoelectric: tabulated, sample shell
- Pair production: Bethe-Heitler
- Electron ionization: Møller sampling
- Electron bremsstrahlung: Seltzer-Berger tables

For neutrons (approximate):
- Elastic: isotropic in CM frame, sample from total cross-section
- Capture: exponential sampling of capture cross-section

**Physics tables on GPU:**
Geant4 pre-computes cross-section tables at startup. Design the export mechanism:
- At `BuildPhysicsTable()`: extract from G4PhysicsTable into flat arrays
- Transfer to GPU device memory (once per run, not per event)
- On GPU: log-log interpolation on energy grid

### Component 4: GPU Geometry Engine (`G4GPUGeometry`)

Two backends — design both, recommend one for NNBAR:

**Backend A: Voxel Grid (recommended for start)**
```
Pre-compute at startup:
    3D grid covering detector bounding box
    Resolution: configurable (e.g. 1mm³ for cosmics, 0.1mm³ for precision)
    Each voxel stores: material_id (uint8), volume_id (uint16), SD_id (int8)

GPU step function:
    current_pos → current voxel → material_id, volume_id
    ray march along direction until voxel changes → step length
    Algorithm: 3DDA (Amanatides & Woo, 1987) — classic, GPU-friendly
```
Estimate GPU memory for NNBAR: calculate voxel count at 1mm³ resolution.

**Backend B: Analytic Geometry**
For NNBAR's simple detector (cylinders + slabs):
```
Ray-cylinder intersection: parametric solution, 2 roots → pick closest
Ray-slab intersection: parallel planes, trivial
Hierarchy: check outermost volume first, recurse inward
```
No approximation, exact boundaries, but harder to generalize to other experiments.

Design the `G4GPUGeometry::Build()` method that reads from Geant4's geometry
at startup and constructs whichever backend was selected.

---

### Component 5: GPU Sensitive Detector (`G4GPUHitBuffer`)

```cpp
struct G4GPUHitBuffer {
    float* edep;           // energy deposit (MeV)
    float* pos_x, *pos_y, *pos_z;
    float* time;
    int*   sd_id;          // which sensitive detector
    int*   track_id;
    int    n_hits;
    int    capacity;
};
```

Hit accumulation on GPU uses atomic operations (`atomicAdd`). At `FlushEvent()`,
hits are read back and injected into Geant4's hit collections via the SD manager.

Design the mapping: GPU `sd_id` → Geant4 `G4VSensitiveDetector*`

## Performance Model

Estimate theoretical performance. Work through the math for NNBAR cosmics:

**Memory bandwidth analysis:**
- Track buffer size: N tracks × (6 floats pos/dir + 1 float energy + 3 ints) = ~40 bytes/track
- At N=256K tracks: 10MB per batch — fits in L2 cache of A100
- Hit buffer: N_hits × 32 bytes — estimate N_hits per cosmic muon event

**Compute analysis:**
- A100: 19.5 TFLOPS FP32
- One muon step: ~50 FP operations (Bethe-Bloch + Highland + random)
- At 256K muon steps per batch: 12.8M FLOP → 0.66 μs on GPU
- Compare to CPU: same 256K steps × ~1 μs/step = 256 ms
- Theoretical speedup (compute bound): ~400,000×

Reality check: it's bandwidth-bound, not compute-bound. Work out the actual estimate
including memory transfers, kernel launch overhead, and host-device sync.

**What speedup is realistic?** Look at Celeritas benchmarks for comparison.

## Validation Framework

Design a validation test suite that runs with EVERY change:

```
For each implemented process, compare GTB vs. full Geant4:

Test 1: Energy deposition profile (muon in lead)
    - Run 10k events with GTB, 10k with Geant4
    - Compare dE/dx vs. depth histogram
    - Accept if KS test p > 0.05

Test 2: Angular distribution (multiple scattering)
    - Compare theta distribution after 10cm iron
    - Accept if mean within 2%, sigma within 5%

Test 3: Hit multiplicity per sub-detector
    - Compare mean hits per event, per SD
    - Accept if within 5%

Test 4: Secondary multiplicity
    - Compare number of secondaries per event
    - Accept if within 10%
```
